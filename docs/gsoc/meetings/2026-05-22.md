# Meeting notes — 2026-05-22 (GP sync)

> Raw working log, kept for shared reference and tracking. Scratch before a stable release. Decisions are distilled into the ADRs (`/docs/adr/`); this is the unpolished record.

Participants: JG, GP, JB.

## Decisions confirmed

- First deliverable = read-only natural-language retrieval from a real archive (not submission).
- Midterm artifact = read + write + enforced HITL, single-agent. Multi-agent stays a post-midterm stretch.
- MCP tools wrap `aiida-restapi`'s Pydantic models in-process (no running REST server) — ADR-02.
- Provider-agnostic LLM layer: lean to `pydantic-ai` over `litellm` — ADR-03.
- Read-only = agent free rein; writes = human confirmation — ADR-08.

## Notes

- Touch base early Aug
- Models differ not just in answer quality but in output format (JSON vs free text) — matters for chaining
- We can run the larger / more expensive models locally, on PSI hardware. Important that the infra is there, however
- Gemma 4 models seems to be well suited for technical tasks, and interaction (e.g., dump JSON that can be consumed by other models)
- 2 parts of testing:
  - flagship cloud models: they already have plenty of knowledge (trained on our docs), and are already very capable themselves, and can do tasks without additional infra
  - small, local models: naturally worse, hence may need additional tooling (e.g. RAG) to achieve tasks
- During the first months, we follow both paths
- We (JG, and MSD group, possibly) refine the system prompts we will integrate, using our scientific domain and AiiDA knowledge
- Clarification: we use the pydantic system in aiida-core, and the aiida-restapi, not an actual running REST API server
- We could use / retain the distinction encoded in the REST API for read / write paths
- We could allow the models to "do whatever they want" for read-only operations, and only ask for human confirmation for write operations
